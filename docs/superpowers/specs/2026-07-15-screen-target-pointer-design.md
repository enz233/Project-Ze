# Screen Target Pointer 屏幕目标指示系统设计

日期：2026-07-15

## 背景

Project-Ze 已具备两个与本设计直接相关的基础能力：

1. `ScreenAnalyzer`：唯一屏幕截图与 Vision 分析服务，用于用户显式发起的屏幕分析。
2. `MoveController`：主进程自动移动控制器，支持按屏幕坐标平滑移动桌宠窗口。

用户希望在屏幕分析时，如果请求中包含“帮我指出 xxx”“在哪里”“帮我找按钮”等目标定位诉求，桌宠不仅能文字说明，还能移动到目标旁边，用“手指指出”差分指向目标位置，并配合气泡提示。

本设计聚焦第一版主动请求能力，并为后续更强视觉指引能力预留边界。普通聊天中的自动触发暂不作为优先路线。

## 目标

1. 在 `.` 开头的显式屏幕分析入口中支持目标指示请求。
2. 通过 Vision/LLM 从当前截图中定位用户所描述的屏幕目标。
3. 将 Vision 返回的截图坐标换算为屏幕坐标。
4. 根据“手指尖锚点”把桌宠窗口移动到目标旁边，使指向差分的手指尖尽量对准目标点。
5. 移动和定位期间使用气泡提示用户不要移动屏幕。
6. 如果屏幕状态发生变化，取消本次指示并明确提示用户重新发起请求。
7. 低置信度、目标不可见、多候选不确定时不移动，避免乱指。
8. 第一版不自动点击、不自动重试、不后台截图。
9. 第三阶段优先增强视觉指引能力，例如方向化指向差分、截图变化检测和候选确认。

## 非目标

1. 第一版不处理普通聊天中的隐式触发，例如用户未使用 `.` 但问“怎么下载这个”。
2. 第一版不做自动点击、自动滚动、自动打开网页或其它替用户操作。
3. 第一版不做持续屏幕监控。
4. 第一版不承诺像 UI 自动化工具一样精确点中控件。
5. 第一版不对当前截图不可见的目标进行定位。
6. 第一版不在屏幕变化后自动重新截图重试。
7. 第一版不实现复杂候选选择 UI。

## 路线判断

原先可考虑三阶段：

1. 主动请求版。
2. 普通聊天中的自然语言触发。
3. 更强视觉指引。

经讨论，优先级调整为：

1. **优先完成第一版主动请求版。** 入口清晰、隐私风险低、误触发少。
2. **随后优先做第三版视觉指引增强。** 这能提升可见体验和定位可靠性，且仍保持用户显式发起。
3. **第二版自然语言触发暂缓。** 普通聊天自动判断存在较难解决的问题：隐私提示、误触发、用户是否真的允许截图、普通问题和屏幕问题边界模糊、以及模型判断不稳定。除非后续有明确 UX 策略，否则不作为近期目标。

## 推荐方案

采用新增编排模块 **ScreenTargetPointer**，不要把所有逻辑塞进 `ScreenAnalyzer`。

```text
用户输入
  ↓
ChatManager / main 判断是否为 "." 屏幕分析请求
  ↓
ScreenTargetPointer
  ├─ 判断是否是目标指示请求
  ├─ 调用 ScreenAnalyzer 截图 + Vision 定位
  ├─ 校验置信度和屏幕稳定性
  ├─ 根据 pointerOffset 换算窗口左上角坐标
  ├─ 调用 MoveController.moveTo()
  ├─ 通知 renderer 切换 point-* 指向差分
  └─ 通过 BubbleManager / BubbleOrchestrator 显示提示气泡
```

职责边界：

- `ScreenAnalyzer`：负责截图和 Vision 分析/定位，不负责移动或视觉表现。
- `ScreenTargetPointer`：负责从用户请求到指示动作的编排。
- `MoveController`：只负责窗口移动、clamp、取消和移动视觉。
- renderer：只负责显示指向差分。
- 气泡系统：负责用户可见提示。

该边界便于后续将目标指示能力复用于其它显式入口，同时避免破坏现有屏幕分析职责。

## 触发规则

第一版只在 `.` 开头屏幕分析入口中启用。

触发示例：

```text
.帮我指出下载按钮在哪
.请指出登录入口
.这个页面哪里能下载，帮我指一下
.帮我找一下设置按钮
```

继续走普通屏幕分析的示例：

```text
.总结这个页面
.这个页面在讲什么
.帮我看看这是什么网站
```

第一版意图判断使用保守关键词规则，不让 LLM 负责复杂触发判断。初始关键词可包括：

```text
指出 / 指给我 / 在哪 / 在哪里 / 帮我找 / 找一下 / 哪个按钮 / 下载在哪 / 怎么点
```

规则命中后进入目标指示流程；未命中则保持现有普通屏幕分析行为。

## Vision 能力边界

本系统定义为“视觉指路助手”，不是精确 UI 自动化。

适合定位：

- 清晰可见的按钮。
- 页面链接。
- 明显文本入口。
- 导航项。
- 当前截图中可见的明显 UI 区域。

不承诺定位：

- 极小图标。
- 多个相似目标中的精确选择。
- 当前屏幕不可见的目标。
- 滚动或切换页面后的旧坐标。
- 被遮挡、过小、模糊或高缩放后的控件。

成功标准应是：对当前屏幕上清晰可见的按钮、文字入口和明显 UI 区域，能够返回大致坐标，并让桌宠移动到旁边指出。

## 定位结果结构

Vision 定位请求必须返回结构化结果，而不是自由文本坐标。

```ts
interface ScreenTargetLocateResult {
  found: boolean;
  label: string;
  confidence: number; // 0~1
  point?: { x: number; y: number }; // 截图坐标系下的目标点
  reason?: string;
}
```

判定规则：

1. `found=false`：不移动。
2. `found=true` 但没有 `point`：不移动。
3. `confidence >= 0.72`：允许移动并指出。
4. `confidence < 0.72`：只气泡提示不确定，不移动。
5. `reason` 用于调试和生成短气泡。

如果后续模型支持更严格的 JSON schema 或结构化输出，应优先使用结构化输出；否则第一版至少要做 JSON 解析和字段校验。

## 坐标换算

截图定位结果中的 `point` 属于截图坐标系，不能直接假设等于屏幕坐标。

截图层或 `ScreenAnalyzer` 需要提供截图帧元信息：

```ts
interface ScreenCaptureFrame {
  imageBase64: string;
  origin: { x: number; y: number };
  size: { width: number; height: number };
  scaleFactor?: number;
}
```

换算规则：

```ts
screenX = frame.origin.x + point.x / (frame.scaleFactor ?? 1);
screenY = frame.origin.y + point.y / (frame.scaleFactor ?? 1);
```

实现前需确认当前截图 API 返回的是物理像素还是 Electron DIP 坐标。设计要求是：定位模块必须明确截图坐标如何映射回屏幕坐标，不能隐式假设。

## 指向锚点

用户期望桌宠移动到目标旁边，并通过“手指指出”差分表现目标位置。因此第一版不应把桌宠中心移动到目标点，而应把桌宠窗口内的“手指尖锚点”对齐到目标屏幕坐标。

```ts
interface PointerPoseConfig {
  pose: 'point-right' | 'point-left' | 'point-up' | 'point-down';
  pointerOffset: { x: number; y: number }; // 手指尖在桌宠窗口内的位置
}
```

换算方式：

```ts
moveTopLeft.x = screenTarget.x - pointerOffset.x;
moveTopLeft.y = screenTarget.y - pointerOffset.y;
```

调用移动：

```ts
moveController.moveTo({
  x: moveTopLeft.x,
  y: moveTopLeft.y,
  anchor: 'top-left',
  reason: 'screen-target-pointer',
});
```

第一版可以先固定一种姿态，例如 `point-right`。如果素材尚未准备好，接口先保留，视觉上临时复用已有差分或静态图。

后续增强优先做方向化站位：

- 目标在屏幕右侧：桌宠站左边，用 `point-right`。
- 目标在屏幕左侧：桌宠站右边，用 `point-left`。
- 目标靠底部：桌宠站上方，用 `point-down`。
- 目标靠顶部：桌宠站下方，用 `point-up`。

## 屏幕稳定性保护

定位开始时先气泡提示：

```text
我看看哦，先别动屏幕~
```

从截图开始到移动完成前，系统进入短暂指示会话：

```ts
type ScreenPointingSessionState =
  | 'capturing'
  | 'locating'
  | 'moving'
  | 'pointing'
  | 'cancelled'
  | 'done';
```

期间如果发现屏幕状态变化，则取消本次指示，不自动重试。

第一版可检测：

- 前台窗口标题变化。
- 前台进程变化。
- 用户发起新的屏幕分析。
- 用户拖拽桌宠。

后续增强可检测：

- 截图指纹变化。
- 明显滚动或页面布局变化。

取消时清理移动和指向视觉，并明确提示：

```text
屏幕变了，我刚才看到的位置可能不准啦。你重新发一次「.帮我指出xxx」吧。
```

可选个性文案：

```text
别乱动屏幕呀，我都看不清了。重新发一次我再帮你指~
```

第一版不自动重试，原因：

1. 避免用户滚动页面时连续触发多次截图和 Vision 调用。
2. 避免坐标基准不断变化，导致桌宠移动体验混乱。
3. 保持用户主导，只有用户明确再次请求才重新读取屏幕。

## 用户体验流程

### 成功流程

用户：

```text
.帮我指出下载按钮在哪
```

桌宠：

```text
我看看哦，先别动屏幕~
```

定位成功后：

- 桌宠移动到目标旁边。
- 指向差分切到 `point-*`。
- 手指尖锚点尽量对齐目标。
- 气泡提示：

```text
我觉得是这里，你看看是不是这个。
```

如果有明确 label：

```text
这里是「Download」按钮。
```

停留 4~6 秒后恢复原状态视觉。

### 找不到目标

```text
我没太看清楚「下载按钮」在哪里。你可以把页面停在目标附近，再让我看一次。
```

不移动。

### 多个候选目标

```text
我看到了几个可能的位置，但不确定你说的是哪一个。你可以说得再具体一点，比如“右上角的下载”。
```

不移动。

### 目标不在当前屏幕

```text
我现在这张屏幕里没看到它，可能需要先滚到对应区域。
```

不移动。

### 屏幕变化

```text
屏幕变了，我刚才看到的位置可能不准啦。你重新发一次「.帮我指出xxx」吧。
```

不移动或取消移动。

### 用户拖拽打断

```text
好啦好啦，我不挡你~
```

取消移动和指向视觉。

### 屏幕边缘目标

如果目标太靠近屏幕边缘，`MoveController` 会 clamp 桌宠窗口，手指尖可能无法完全对齐目标。第一版接受该限制，并可提示：

```text
它在屏幕边边这里，我尽量指过去啦。
```

## 错误处理原则

1. Vision 调用失败：保持屏幕分析失败提示，不移动。
2. JSON 解析失败：当作定位失败，不移动，记录日志。
3. 坐标缺失或非法：不移动，记录日志。
4. 坐标在屏幕外：交给 `MoveController` clamp；如果 clamp 导致指尖无法对齐，用屏幕边缘文案提示。
5. 移动被取消：清理指向视觉，不继续显示定位成功文案。
6. 置信度不足：宁可不动，也不要乱指。
7. 用户发起新屏幕分析：取消旧会话，以新请求为准。

## 后续增强优先级

### 阶段 1：主动请求版

- 主入口为 `.`。
- 保守关键词判断。
- Vision 结构化定位。
- 高置信度移动并指出。
- 屏幕变化取消，不重试。
- 只移动和提示，不点击。

### 阶段 2：视觉指引增强

优先于自然语言自动触发。

- 根据目标位置自动选择 `point-left/right/up/down`。
- 增加独立指向差分素材。
- 支持截图指纹变化检测。
- 支持候选目标确认。
- 支持“你说的是这个吗？”式交互确认。
- 支持更自然的停留、返回和恢复状态表现。

### 暂缓：普通聊天自然语言触发

暂缓原因：

- 普通聊天中自动截图涉及隐私边界。
- 用户意图可能只是问概念，不一定希望读取屏幕。
- LLM 判断“是否需要看屏幕”可能不稳定。
- 误触发会造成打扰和额外成本。
- 需要更完整的 UX 策略，例如显式授权、提示、取消和历史偏好。

除非后续明确设计隐私和授权策略，否则不进入近期实现计划。

## 测试与验证

1. `.总结这个页面` 仍走普通屏幕分析。
2. `.帮我指出下载按钮在哪` 进入目标指示流程。
3. Vision 返回高置信度坐标时，桌宠移动到目标旁边。
4. 指向锚点换算后，手指尖大致对准目标点。
5. 置信度低时不移动。
6. 目标不可见时不移动。
7. 多候选不确定时不移动并提示用户说得更具体。
8. 屏幕变化时取消并明确提示重新请求。
9. 用户拖拽桌宠时取消移动。
10. 移动结束后恢复原状态视觉。
11. 坐标接近屏幕边缘时不会把桌宠移出可用区域。
12. TypeScript 编译通过。

## 成功标准

- 项目存在独立的屏幕目标指示设计边界。
- 第一版只在 `.` 主动请求中触发，不引入后台截图。
- Vision 定位结果经过结构化校验和置信度门槛。
- 桌宠通过手指尖锚点对齐目标，而不是挡在目标中心。
- 屏幕变化时明确取消并提示用户重新请求。
- 失败分支均不移动，避免错误指示。
- 后续增强优先投入视觉指引，而不是普通聊天自动触发。

## 自检

- 无占位符。
- 第一版范围聚焦：主动请求、定位、移动、指向和提示。
- 普通聊天自然语言触发已明确暂缓，避免近期范围失控。
- 模块边界清晰：`ScreenAnalyzer` 管截图和定位，`ScreenTargetPointer` 管编排，`MoveController` 管移动，renderer 管视觉。
- 坐标换算、指向锚点、屏幕稳定性和失败分支均有明确规则。
