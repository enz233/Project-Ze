# 聊天气泡长回复拆分设计

日期：2026-07-16

## 背景

Project-Ze 的聊天回复通过聊天模型生成 `<item>...</item>`，再由 `ChatManager` 解析后逐条发送到 renderer 的 `show-bubble`。当前实现已经有一层长文本拆分：单个 item 超过 30 字符时按中文标点或固定长度拆分，并以约 1.5-2.5 秒间隔显示多个气泡。

这个机制能避免单个气泡过长，但体验偏机械：阈值过短，容易把一句自然话切得太碎；步骤、清单或操作说明不会稳定按“一步一个气泡”呈现；模型如果把多个步骤塞进一个 `<item>`，本地兜底只能按字符长度拆，语义边界不够清楚。

用户希望聊天回复更自然：普通长回答应切成多个自然气泡；如果回答包含步骤，则一步一切。

## 目标

1. 让普通聊天长回答按自然语义切分成多个短气泡。
2. 让步骤类回答稳定按“一步一个气泡”显示。
3. 保持桌宠陪伴感：每个气泡像一句自然说话，而不是长段落或机械截断。
4. 优先通过提示词让模型主动输出合适的 `<item>`，再用本地拆分兜底。
5. 将改动集中在聊天回复格式和 `ChatManager` 拆分逻辑，不改 renderer 气泡布局。
6. 保留现有聊天历史、TTS fallback、Response Workflow 统一聊天输出链路。

## 非目标

1. 不新增可滚动长文本面板。
2. 不改 renderer 的气泡位置、样式或 3 秒自动隐藏机制。
3. 不改变 TTS 播放模型，只调整传入 TTS/气泡的文本分段。
4. 不让普通聊天变成长篇教程；模型仍应优先短句、1-3 个自然气泡。
5. 不把屏幕分析或工作流结果改成单独 UI；它们继续通过统一聊天气泡输出。

## 推荐方案

采用“提示词约束 + 本地语义兜底拆分”的混合方案。

模型侧通过回复格式提示词明确：

- 每个 `<item>` 代表一个自然语义单元。
- 普通聊天使用短句，避免一个 `<item>` 放长段落。
- 如果回答包含步骤、清单或操作顺序，则每一步单独一个 `<item>`。
- 不要把多个步骤塞进同一个 `<item>`。

本地侧在 `ChatManager` 中替换现有固定 30 字符拆分策略，新增更语义化的 `splitBubbleText` 兜底：

1. 优先识别步骤边界。
2. 再识别换行和段落边界。
3. 再按句末标点拆分。
4. 再按逗号、顿号等弱标点拆分。
5. 最后才做最大长度硬切。

选择该方案的原因：模型配合时输出最自然；模型偶尔不按要求时，本地仍能保证气泡不过长并尽量按语义拆分。实现范围小，风险低。

## 拆分规则

### 输入来源

拆分输入来自两类文本：

1. 模型正确输出的多个 `<item>` 内容。
2. 模型未输出 `<item>` 时的完整 fallback 文本。

两类输入都经过同一套 `splitBubbleText` 兜底，以保证行为一致。

### 步骤边界

识别以下常见步骤格式：

- `1. ...` / `1、...` / `1）...` / `（1）...`
- `步骤一：...` / `步骤 1：...`
- `第一步：...` / `第二步：...`
- 行首 `- ...` / `• ...` 这类清单项

当检测到多个步骤项时，优先按步骤项切分。每一步默认生成一个气泡。若某一步本身过长，再在该步骤内部按句子拆分，但应尽量保留步骤编号或“第一步”前缀在第一段中。

### 普通语义边界

非步骤文本按以下优先级拆分：

1. 换行或段落。
2. 句末强标点：`。`、`！`、`？`、`!`、`?`、`；`、`;`。
3. 弱标点：`，`、`,`、`、`、`：`、`:`。
4. 最大长度硬切。

硬切只作为兜底，用于没有标点的超长字符串。

### 长度建议

- 目标长度：约 36-48 个中文字符。
- 最大兜底长度：约 64 个中文字符。
- 已经短而自然的 `<item>` 不再拆分。

英文和中英混合文本不做复杂 token 估算，仍按 JavaScript 字符长度近似处理。后续如果英文长句体验不好，再单独引入更精细的宽度估算。

## 数据流

```text
AI full response
  ↓
parse <item>
  ↓
raw text items
  ↓
splitBubbleText(item)
  - step/list boundary
  - paragraph boundary
  - sentence boundary
  - weak punctuation boundary
  - hard cut fallback
  ↓
texts[]
  ↓
TTS speakAll(texts) or showTextSequence(texts)
  ↓
renderer show-bubble
```

## 模块边界

| 模块 | 负责 | 不负责 |
|---|---|---|
| `ChatManager` | 解析 `<item>`、长回复拆分、按序发送气泡、传入 TTS 分段 | 气泡视觉布局 |
| `RESPONSE_FORMAT_PROMPT` | 引导模型主动短句、多气泡、步骤一泡一步 | 绝对保证模型遵守 |
| renderer `showBubble` | 展示单个气泡、自动隐藏、字幕期间防覆盖 | 文本语义拆分 |
| `TTSManager` | 按传入文本序列播放 | 决定如何拆分文本 |

## 错误与边界处理

1. 模型未输出 `<item>`：使用完整文本作为一个 raw item，再进入本地拆分。
2. `<item>` 为空：忽略空 item。
3. 步骤识别误判：只有检测到至少两个步骤边界时才启用“步骤优先拆分”，减少普通句子被误拆。
4. 超长无标点文本：按最大长度硬切，避免气泡溢出。
5. TTS 开启：TTS 使用拆分后的文本序列；单段仍保留现有最多 200 字符截断兜底。
6. 字幕播放期间：renderer 现有 `subtitleActive` 仍会阻止普通气泡覆盖，不在本次改动中改变。

## 测试计划

新增或更新契约测试，至少覆盖：

1. 普通长句按句末标点拆成自然气泡。
2. 已经合理的多个 `<item>` 不被过度拆碎。
3. `1. ... 2. ...` 按一步一泡拆分。
4. `步骤一/步骤二` 或 `第一步/第二步` 按一步一泡拆分。
5. 行首 `- ...` 清单项按项拆分。
6. 超长无标点文本按最大长度兜底硬切。
7. fallback 文本没有 `<item>` 时仍能拆分。
8. TTS 路径和非 TTS 路径都使用同一拆分结果。

3. 如新增独立说明或测试契约，可补充到 `docs/` 下对应文档。

## 验收标准

1. 普通长回答不再按固定 30 字符机械切碎。
2. 步骤回答能稳定呈现为“一步一个气泡”。
3. 单个气泡不会出现明显过长段落。
4. 普通聊天仍保持短句和陪伴感。
5. 现有 `<item>` 回复格式、TTS fallback、聊天历史和 Response Workflow 输出链路不被破坏。
6. 相关契约测试通过。
